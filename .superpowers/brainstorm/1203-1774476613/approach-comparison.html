Approach 2 vs 3: Architecture Deep Dive

Focus on how each handles real scenarios — and where each breaks down.

Approach 2: Layered Resolution

Existing pipeline stays intact. New SublocationResolver layer activates on arrival at an address.

System Architecture
How It Handles Scenarios
Dave's work day
1. Existing pipeline: Sleep→Travel→Work→Travel→Home
2. PersonBehavior: arrive at Office
3. → handoff to SublocationResolver
4. Resolver: Work task + Office graph + tags
5. Produces: Lobby→Elevator→Cubicle→BreakRoom→...
6. SublocationBehavior: execute subtasks, leave traces
✓ Works well. Clean handoff.
Killer breaks into home
1. Existing pipeline: KillPerson task → victim's address
2. PersonBehavior: arrive at victim's home
3. → handoff to SublocationResolver
4. Resolver: KillPerson task → "Intrude" behavior
5. Find [covert_entry] → Back Window
6. Find room with victim → Bedroom
7. Pathfind: Window→Hallway→Bedroom
8. Execute kill action in Bedroom
✓ Works. Resolver handles the strategy.
Heist planner calls getaway driver
1. Planner's existing pipeline: has "CallPerson" task
2. SublocationResolver: find [phone] → Kitchen
3. Planner walks to Kitchen, starts call
4. Now need to interrupt the Driver...
5. Driver is at a DIFFERENT address
6. Driver's PersonBehavior is in the existing layer
7. SublocationResolver can't reach up to trigger
8. an interrupt at the address level
9. Need: SublocationBehavior → PersonBehavior
10. reverse communication (layer violation)
⚠ Layer boundary makes cross-address interaction awkward. Sublocation events need to "reach up" through the layer.
NPC realizes they need to leave early
1. Dave is at work (sublocation layer active)
2. Gets a phone call: "emergency at home"
3. Sublocation layer wants to end the
4. Work task early and trigger travel
5. But travel is owned by the address layer
6. Sub layer must signal up: "I'm done early"
7. Address layer must: rebuild schedule?
8. Or inject an emergency travel task?
9. Two systems negotiating control.
⚠ Upward pressure from sublocation events to address-level schedule changes is inherently awkward in a layered model.
Verdict
Works great for contained activities — arrive, do stuff in rooms, leave. Struggles when sublocation-level events need to affect address-level behavior. The layer boundary becomes a wall you keep drilling holes through.

Approach 3: Recursive Tasks

One unified system. Tasks decompose into sub-tasks at any depth. Same resolution logic at every scale.

System Architecture
How It Handles Scenarios
Dave's work day
1. Objective "MaintainJob" → Work task (P:20)
2. TaskResolver: Work task decomposes into:
   Enter[entrance] → Goto[work_area] → Exit[entrance]
3. Goto[work_area] decomposes via pathfinding:
   Move→Lobby → Move→Elevator → Move→Cubicle
4. Periodic sub-tasks [food] and [restroom]
   preempt by priority when timers fire
5. TaskExecutor walks the tree, traces follow
✓ Same result as Approach 2, one system.
Killer breaks into home
1. Objective "CommitMurder" → KillPerson task (P:40)
2. TaskResolver decomposes with "Intrude" behavior:
   Enter[covert_entry] → Find[victim] → Execute
3. Enter[covert_entry] → Back Window
4. Find[victim] → Bedroom (person query, not tag)
5. Pathfind: Window→Hallway→Bedroom
6. Execute kill action
✓ Same mechanism. "Intrude" is just a decomposition strategy.
Heist planner calls getaway driver
1. Planner's task tree: CallPerson(Driver) at P:35
2. Decomposes: Goto[phone] → MakeCall
3. Planner walks to Kitchen phone
4. MakeCall executes → injects interrupt
5. Driver gets: AnswerPhone task at P:30
6. Same system! P:30 > Driver's current Sleep P:20
7. Driver's TaskExecutor preempts Sleep
8. Decomposes: Goto[phone] → AnswerCall
9. Driver walks Bedroom→Kitchen, picks up
10. Call ends → P:30 task done → Sleep resumes
11. Driver walks Kitchen→Bedroom
✓ No layer violation. Interrupts are just high-priority tasks injected into any NPC's tree. Same mechanism at every level.
NPC leaves work early
1. Dave at work, phone rings (interrupt P:35)
2. AnswerPhone → learns of emergency
3. New task: GoHome(emergency) at P:38
4. P:38 > Work P:20 → preempts entire work tree
5. Decomposes: Exit[entrance] → Travel → Home
6. No schedule rebuild needed.
7. Priority just wins. Same as how Sleep
8. beats Idle in the existing system.
✓ Priority-based preemption works at every scale. No "layers negotiating control."

Concrete Risks: Where Each Approach Breaks Down

Approach 2 Risks

Layer boundary becomes porous
Every cross-address interaction (phone calls, NPCs reacting to events at other locations, emergencies) requires drilling through the layer boundary. Over time you accumulate callbacks, events, and upward signals until the "clean separation" is mostly ceremony — two systems tightly coupled through a growing API surface.
Likelihood: High — your game design inherently needs cross-scale interactions.
Two scheduling systems to maintain
ScheduleBuilder handles address-level time slots. SublocationResolver handles room-level sequencing within those slots. Bugs where they disagree (sublocation sequence takes longer than the address-level time slot allows) are subtle and hard to reproduce.
Likelihood: Medium — timing edge cases at the boundary.
Migration tax later
If you eventually want recursive tasks (and the scenarios above suggest you will), the layered system becomes throwaway scaffolding. The SublocationResolver, SublocationBehavior, and the handoff protocol all get replaced.
Likelihood: Medium-High — depends on how far you push NPC interactions.

Approach 3 Risks

Refactor scope
ObjectiveResolver, ScheduleBuilder, PersonBehavior, and SimTask all change significantly. The current ScheduleBuilder's minute-by-minute priority resolution is clever but flat — it assumes tasks don't nest. Replacing it with a tree-walking executor is a meaningful rewrite, not a refactor.
Mitigation: The existing codebase is small (~150 lines each). Rewriting is feasible, not scary.
Debugging recursive decomposition
"Why did the NPC walk to the bathroom at 3 AM?" requires tracing through multiple decomposition levels. The task tree can get deep, and emergent behavior from priority preemption can be surprising.
Mitigation: Build a task tree inspector into the Person Inspector UI from day one. The tree IS the explanation — if you can see it, you can debug it.
Over-abstraction risk
"Everything is a task" can become "everything is a generic blob." If the TaskResolver becomes a god class that handles every decomposition strategy, you've traded one problem for another.
Mitigation: Decomposition strategies (WorkDay, Intrude, Patronize) are separate classes/functions. TaskResolver dispatches, doesn't contain logic.
