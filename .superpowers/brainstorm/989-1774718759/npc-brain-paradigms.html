NPC Brain Paradigms — Beyond Scheduling

What does an NPC brain actually need to do? Let's start from requirements, not from the existing code.

What does the game actually need from NPCs?

Player-facing needs
  • Where is this person right now? (stakeout, tailing)
  • What traces did they leave? (investigation)
  • Can I interact with them? (conversation, arrest)
  • What did witnesses see? (who was where when)
Simulation needs
  • NPCs react to events (heist, arrest, player presence)
  • NPCs coordinate (groups form, act, disband)
  • New NPCs spawn with plausible history (backfill)
  • Actions produce traces proportional to drama, not granularity
  • Time can fast-forward without simulating every minute

Key insight: The player never observes minute-by-minute NPC movement. They observe presence (stakeout), traces (investigation), and availability (interaction). The simulation's job is to produce those three outputs believably — not to compute a perfect timeline.

Paradigm 1: Pre-computed Schedule (what we have now)

Plan the whole day, then replay it. Interrupts = rebuild the schedule.

Mental model
Each NPC has a 1440-slot daily schedule, pre-computed from objectives. Every frame: look up "what should I be doing at this minute?" and do it. Interrupt happens? Set NeedsScheduleRebuild = true. Next frame: throw away schedule, recompute from scratch. ┌─────────────┐ │ Objectives │ └──────┬───────┘ ▼ ┌─────────────┐ │ Resolve to │ │ Tasks │ └──────┬───────┘ ▼ ┌─────────────┐ │ Build 1440- │──→ minute 0: sleep at home/bedroom │ slot schedule│──→ minute 480: travel to work └──────┬───────┘──→ minute 510: work at office/cubicle ▼ ... ┌─────────────┐ │ Execute │ │ per frame │ └──────────────┘
How it handles the hard cases
Driver arrested mid-heist All other NPCs' schedules are stale — they still think Driver is coming. Need manual invalidation of every linked schedule. How does the heist template know to abort? It doesn't — schedules are independent.
Bank patrons react to heist Each patron needs NeedsScheduleRebuild. Their new schedule: "cower, then wait for police, then go home." But their original schedule is gone — do they go back to work after? When? The rebuild doesn't know their pre-interrupt intent.
Fast-forward 6 hours Works well — just advance time, schedule entries still valid. ✅
Backfill NPC history Generate past schedules, walk through them, deposit traces. Expensive — you're replaying days of 1440-slot schedules to produce a handful of traces.
Groups Requires pre-coordinated schedules across NPCs. Fragile — any deviation breaks synchronization.

Paradigm 2: Intent Stack — "What do I want to do right now?"

NPCs have a routine and react to events. No pre-computed schedule. Behavior emerges from prioritized intents.

Mental model
Each NPC has: - A Routine: repeating time blocks ("home 10pm-8am", "work 9am-5pm", "diner 12-1pm") - An Intent Stack: things overriding the routine, newest on top Every tick: top of intent stack (or routine if stack empty) → determines current location + activity + trace generation. Interrupt? Push a new intent. When it completes, pop it — previous intent (or routine) resumes naturally. ┌───────────────────────────────┐ │ Intent Stack │ │ ┌─────────────────────────┐ │ │ │ "react to heist" (p:60) │ │ ← newest: pushed by event │ ├─────────────────────────┤ │ │ │ "eat lunch at diner" │ │ ← was doing this, interrupted │ ├─────────────────────────┤ │ │ │ (routine: work 9-5) │ │ ← resumes after lunch │ └─────────────────────────┘ │ └───────────────────────────────┘ Each intent knows: - What activity am I doing? - Where should I be? - When am I done? (or: what condition ends this?) - What traces do I leave while doing this?
Heist scenario under Intent Stack
8:30 — Crime template pushes "participate_in_heist" intent onto each member. Each member's routine is now overridden. 9:00 — Each member's intent says "go to planner's house." They travel there independently. On arrival, they're "assembled." 9:15 — Group forms. A GroupIntent is pushed onto all members: "execute heist plan as group" The group intent manages the shared timeline: → drive to alley (group moves together) → wait until 11:00 → split: each member gets a sub-intent for their role → regroup: sub-intents complete, group resumes 11:10 — Group drives to lot. GroupIntent disbands. "participate_in_heist" intent pops. Each member's routine resumes (or next solo intent kicks in). 11:25 — Driver has a queued intent: "stash money at warehouse." 18:30 — Bagman has a timed intent: "pick up from warehouse closet." DRIVER ARRESTED AT 10:45? → Driver's intent stack is replaced with "arrested" intent. → Group detects missing member at regroup. → Group intent has a contingency: abort? continue without driver? → Other members get "flee" or "improvise getaway" intent pushed. → No schedule rebuild needed. Each NPC reacts to their current state. BANK PATRON DURING HEIST? → Patron is in the bank. Heist event pushes "react_to_threat" intent. → Patron cowers for duration. When heist ends, intent pops. → "Questioned by police" intent pushed (takes 30-60 min). → When that pops, patron's routine resumes — they go back to work or go home depending on what time it is. → No schedule rebuild. Routine was never lost.
How it handles the hard cases
Driver arrested mid-heist Driver gets "arrested" intent. Group detects it at next sync point. Group intent branches to contingency. Other members react individually. No schedule invalidation — intents handle it.
Bank patrons react to heist "React to threat" intent pushed onto everyone present. When resolved, it pops and routine resumes. Patron naturally goes back to whatever their routine says for this time of day. Previous state is preserved on the stack.
Fast-forward 6 hours Trickier — need to resolve intents forward. But most NPCs are on routine, which is trivial to fast-forward. Only NPCs with active non-routine intents need step-through. Can batch routine NPCs.
Backfill NPC history Routine IS the history. "This person works at the diner 9-5 and lives at 45 Oak St" → generate traces from the routine pattern directly, no simulation needed. Only special events need explicit trace generation.
Groups GroupIntent coordinates naturally — it's just an intent shared by multiple NPCs. Split = push sub-intents, regroup = wait for sub-intents to pop.

Pros

  • Interrupts are natural — push/pop, no rebuild
  • Routine never lost — it's always at the bottom of the stack
  • Groups are just shared intents, not a separate system
  • Backfill is cheap — routines generate traces directly
  • Reactive by default — any event can push an intent
  • Composable action templates map directly to intents

Cons

  • Fast-forward needs care (but routine NPCs are trivial)
  • Debugging "why is this NPC here?" means reading the stack
  • New paradigm — significant rewrite of scheduling system

Paradigm 3: Event-Sourced — "What happened to me?"

NPC state is derived from a log of events. No mutable schedule or intent stack. History IS the data model.

Mental model
Each NPC has: - A Routine: same as Paradigm 2 - An Event Log: immutable list of things that happened to/by this NPC - Current state is DERIVED: replay log from last checkpoint to get current location, activity, relationships, etc. Something happens? Append an event to the log. "Where is this NPC?" → derive from routine + events. "What traces exist?" → traces ARE events in the log. Timeline: ──●────────●────────●──────●────────●────→ spawn hired date heist arrest at diner w/X starts Events: { type: "hired", job: diner_cook, time: day_1 } { type: "date", with: person_X, at: person_X.home, time: day_3_19:00 } { type: "joined_group", group: heist_crew, time: day_5_09:00 } { type: "arrested", time: day_5_10:45 } Current state at any point in time = routine + events before that time
How it handles the hard cases
Driver arrested mid-heist "Arrested" event appended. Group queries Driver's state, sees arrest event, appends contingency events for other members.
Bank patrons react to heist "Witnessed heist" event for each patron. Derive behavior: cowering until "heist_ended" event, then "police_interview" event, then routine resumes.
Fast-forward 6 hours No simulation needed — just advance clock. State derived on demand.
Backfill NPC history Perfect fit — just insert historical events into the log. ✅✅
Groups Group events in the log. But coordinating real-time group behavior requires deriving state constantly, which is expensive.

Pros

  • Backfill is trivially natural — insert past events
  • History is first-class — "what happened" is the data model
  • Fast-forward is free — no simulation, derive on demand
  • Traces and events are unified — no separate systems
  • Time-travel debugging: look at any point in an NPC's history

Cons

  • "What is this NPC doing RIGHT NOW?" requires derivation every query
  • Real-time behavior (groups, reactions) needs a separate execution layer on top
  • Risk of becoming two systems: event log for history + something else for real-time
  • Performance: deriving state from log grows expensive over time (needs checkpointing)
  • Most complex to implement correctly

Recommendation: Paradigm 2 (Intent Stack) with Event Log for History

The Intent Stack gives you reactive, interruptible NPC behavior with natural group support. The routine at the bottom of the stack means you never lose "normal life" — it's always there to resume.

Borrow the best idea from Paradigm 3: keep an event log for history. When you backfill an NPC, you're generating events, not replaying schedules. Traces are generated from events. But the real-time brain is the intent stack — it decides what the NPC is doing now and what they'll do next.

This also maps cleanly onto the composable action template system (Approach C from earlier): an action template IS an intent. BreakInMurder is an intent that pushes sub-intents (ForcedEntry, Fight, Kill). A group heist is a shared intent that pushes individual role intents during the split phase.

Routine ──→ "where would I normally be at this time?" │ Intent Stack ──→ "but what am I ACTUALLY doing right now?" │ Action Templates ──→ "HOW do I do it, and what traces do I leave?" │ Event Log ──→ "what DID I do?" (history, backfill, investigation)
