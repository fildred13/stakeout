Simulation Overhaul — Project Sequence

Capturing all brainstorming decisions + a sequenced breakdown of projects. Review each section.

Design Decisions Made

What we've agreed on so far — this becomes the "constitution" for all sub-projects.

Area Decision
Location hierarchy World → City → Address → Location → (optional) Sub-Location. Strict 2-level nesting below Address, not recursive. Tags (exterior, publicly_accessible, etc.) handle behavioral variations. Composable location types (ExteriorParkingLot, ApartmentUnit, etc.) bring their own sub-locations, tags, and rules.
Doors & access points Doors are real objects with mutable state (locked/unlocked, broken/repaired), but they're properties of the Location or Sub-Location they guard — not edges in a graph. Address type templates define what doors, windows, etc. exist. No more BFS pathfinding or sublocation graph traversal.
Fixtures vs traces Separate systems, merged at query time for investigation. Fixtures are stable elements defined by templates (computer, mailbox, safe). Traces are dynamic evidence from actions/events (fingerprints, blood, receipts). Fixtures can contain traces (computer contains emails, trash can contains receipts). Investigation discovers both.
Action templates Hybrid fluent builder (Approach C). Composable C# event primitives (ForcedEntry, Fight, MoveTo, etc.) composed via a readable fluent API (ActionSequence). Templates range from simple (go for a run) to complex (heist). Unified system — no distinction between "routine action" and "crime action" at the engine level.
NPC brain Intent Stack paradigm. NPCs have obligations (job shifts, sleep), drives (traits + state → desired actions), and defaults (be at home). These resolve into intents. Active intents override lower-priority ones via push/pop. Routine is never lost — it's always at the bottom. No pre-computed 1440-slot schedule.
Groups Explicit Group entity with shared intent. Members join/leave. Group intent manages shared timeline with Split/Regroup for parallel actions. Individual intents suspend while grouped, resume on disband. Same system handles a 2-person date and a 4-person heist.
Trace generation Declarative, not physical. Residency → bulk fingerprints/personal items. Action templates emit traces as part of execution. No per-door fingerprint deposition from traversal. Traces can decay over time.
History & backfill Event log records what happened (for investigation, debugging, history). Routine-based backfill for general population (cheap — derive traces from routine pattern). Event-level backfill for crime-related NPCs. No replaying minute-by-minute schedules.
Businesses First-class entities with operating hours, shift definitions, and staffing needs. NPC generation fills shifts. NPCs show up via job obligation.
Performance model Hundreds/low thousands of NPCs. Routine NPCs are free to fast-forward (routine is a function: time → location, no per-tick simulation). Only NPCs with active non-routine intents need step-through resolution. Stakeout is an event stream with trigger conditions.

Project Sequence

Each project builds on the previous. Each gets its own design spec → implementation plan → implementation cycle.

Project 1

Location Hierarchy & Address Templates

Replace the sublocation graph with the new Address → Location → Sub-Location hierarchy. Define the template system for composable address types. Implement doors/access points as properties of locations. Migrate existing sublocation generators to the new model.

Delivers
  • Address, Location, SubLocation entities with tag system
  • Door/AccessPoint as mutable objects on locations
  • Address type template system (composable location types)
  • Migrated generators: SuburbanHome, ApartmentBuilding, Diner, Office, Park, DiveBar
  • Remove SublocationGraph, SublocationConnection, BFS pathfinding
Dependencies

None — this is the foundation everything else builds on.

↓
Project 2

Fixtures & Trace System

Build the fixture system (stable objects defined by templates) and redesign the trace system (dynamic evidence from events). Fixtures can contain traces. Define the investigation query interface that merges both for player-facing discovery.

Delivers
  • Fixture entity with container semantics (can hold traces)
  • Fixture definitions as part of address type templates
  • Redesigned Trace entity (broader than fingerprints — marks, items, sightings, records)
  • Trace decay system
  • TraceEmitter API for action templates to deposit traces
  • Investigation query: merged fixture + trace discovery with difficulty weighting
  • Remove FingerprintService (replaced by TraceEmitter)
Dependencies

Project 1 (fixtures live on locations; traces reference locations).

↓
Project 3

Business Entities & Staffing

Businesses as first-class entities with operating hours, shift definitions, and staffing requirements. NPC generation fills shifts. Lays the groundwork for job obligations in the NPC brain.

Delivers
  • Business entity (operating hours, shift slots, staffing rules)
  • Business type templates (diner, bar, office, bank, etc.)
  • NPC-to-shift assignment during generation
  • Business ↔ Address linkage (a diner occupies an address)
  • Day-of-week awareness for shifts
Dependencies

Project 1 (businesses occupy addresses).

↓
Project 4

NPC Brain — Intent Stack & Routine Resolver

The core NPC brain rewrite. Replace the schedule system with the intent stack. Implement the routine resolver (obligations + drives + defaults). NPCs decide what to do based on who they are, not a pre-computed schedule.

Delivers
  • Intent stack (push/pop, priority-based interruption)
  • Routine resolver: obligations (job shifts, sleep) + drives (traits) + defaults
  • Intent → location resolution ("I want to work" → "I should be at the office")
  • NPC travel between addresses (triggered by intent, not pre-scheduled)
  • Fast-forward: routine NPCs as time→location function, intent NPCs step-through
  • Event log (record what happened for history/investigation)
  • Replace: ScheduleBuilder, ObjectiveResolver, DailySchedule, PersonBehavior
Dependencies

Project 1 (locations), Project 3 (job obligations need business shifts).

↓
Project 5

Action Template Engine

The fluent builder system for composable action templates. Event primitives (ForcedEntry, Fight, MoveTo, etc.), ActionSequence builder, TraceEmitter integration. Templates produce intents and traces.

Delivers
  • ActionSequence fluent builder (Do, MoveTo, Maybe, If, OneOf)
  • ActionContext (state bag threaded through events)
  • Event primitives: ForcedEntry, Movement, Violence, Social, Search
  • Integration with TraceEmitter (actions produce traces)
  • Integration with intent stack (templates resolve as intents)
  • Initial templates: DailyRoutine actions (go to work, sleep, eat, idle)
  • Replace: ActionExecutor, decomposition strategies
Dependencies

Project 2 (trace emitter), Project 4 (intents drive template execution).

↓
Project 6

Groups & Coordinated Actions

Group entity with shared intent, join/leave lifecycle, Split/Regroup for parallel actions. Enables multi-NPC coordinated activities from dates to heists.

Delivers
  • Group entity (members, shared state, shared vehicle/loot)
  • GroupIntent (shared intent pushed onto all members)
  • Join/Leave/Disband lifecycle
  • Split/Regroup for parallel sub-actions
  • Member interruption handling (what if someone gets arrested?)
  • Group action templates: DateNight, CriminalMeeting, Heist (initial set)
Dependencies

Project 4 (intent stack), Project 5 (action templates).

↓
Project 7

History Backfill & NPC Spawning

When new NPCs enter the simulation, generate a plausible past. Routine-based trace backfill for general population. Event-level backfill for crime-related NPCs with planted history.

Delivers
  • Routine-based backfill (derive traces from routine pattern over N days)
  • Event-level backfill for crime NPCs (planted meetings, purchases, etc.)
  • Backfill trace generation (residency fingerprints, workplace traces, etc.)
  • Late-spawning NPC integration (new NPCs enter mid-game with history)
Dependencies

Project 2 (traces), Project 4 (routines), Project 5 (action templates for event backfill).

↓
Project 8

Crime Templates & Scenarios

Rebuild the crime system on top of the new engine. Crime scenarios use action templates, groups, trace generation, and backfill to create rich investigatable crimes. Start with murder, expand to burglary, conspiracy, etc.

Delivers
  • Crime scenario system (replaces CrimeGenerator + ICrimeTemplate)
  • Murder scenario (with variable methods, trace generation, witness events)
  • Crime NPC casting (pick perpetrators, victims, accomplices from population)
  • Drive injection (crime gives NPCs new drives/intents)
  • Integration with groups for multi-person crimes
Dependencies

All previous projects — this is the capstone that exercises everything.

Notes on Sequencing

  • Projects 1-3 are foundational and can be done with the existing scheduling system still running. The game doesn't break until we swap out the NPC brain in Project 4.
  • Project 3 (Businesses) could run in parallel with Project 2 (Fixtures/Traces) — they don't depend on each other, only on Project 1.
  • Project 4 is the big bang — this is where the old scheduling system gets replaced. Projects 1-3 prepare the ground so this can be as focused as possible.
  • Projects 5-6 build on the new brain — action templates and groups are layered on top of the intent stack.
  • Projects 7-8 are the payoff — backfill and crime templates exercise the full system.
  • Player-facing UI (the apartment building walkthrough screens, investigation mechanic, stakeout UI) is a separate workstream that can happen in parallel once Project 1 establishes the new location hierarchy.
