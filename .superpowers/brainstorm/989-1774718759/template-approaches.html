Action Template Approaches

Same scenario (break-in murder) shown three ways. Which feels right?

Approach A: Pure C# — Composable Event Builder

Events are small C# methods. Templates compose them with control flow. All code, no config.

File structure
src/simulation/actions/ events/ # Reusable event primitives ForcedEntry.cs # BreakWindow, PickLock, BreakDoor Movement.cs # MoveTo, ChaseFromTo, FleeFromTo Violence.cs # Fight, Kill, Stab, Strangle Social.cs # HangOut, Converse, HaveSex Search.cs # Ransack, SearchFor, StashItem templates/ # Composed action sequences BreakInMurderTemplate.cs BurglaryTemplate.cs DateVisitTemplate.cs DailyRoutineTemplate.cs ActionContext.cs # State bag passed through events TraceEmitter.cs # Events call this to deposit traces
events/ForcedEntry.cs
public static class ForcedEntry { public static void Execute(ActionContext ctx) { // Pick method based on location properties + randomness var method = ChooseMethod(ctx); switch (method) { case EntryMethod.Window: ctx.Traces.Add(Trace.Mark("broken window", ctx.EntryPoint)); ctx.Traces.Add(Trace.Fingerprint(ctx.Actor, ctx.EntryPoint, "window exterior")); ctx.Traces.Add(Trace.Fingerprint(ctx.Actor, ctx.EntryPoint, "window frame")); if (ctx.Random.Chance(0.3f)) ctx.Traces.Add(Trace.Mark("torn clothing fiber on window frame", ctx.EntryPoint)); ctx.Noise += NoiseLevel.Loud; break; case EntryMethod.LockPick: ctx.Traces.Add(Trace.Mark("scratch marks on lock", ctx.EntryPoint)); ctx.Traces.Add(Trace.Fingerprint(ctx.Actor, ctx.EntryPoint, "door handle")); ctx.Noise += NoiseLevel.Quiet; break; case EntryMethod.BreakDoor: ctx.Traces.Add(Trace.Mark("door broken down", ctx.EntryPoint)); ctx.Traces.Add(Trace.Footprint(ctx.Actor, ctx.EntryPoint, "boot print on door")); ctx.Noise += NoiseLevel.VeryLoud; break; } // Neighbors might witness based on noise if (ctx.Noise >= NoiseLevel.Loud) Witnesses.CheckNeighbors(ctx, "heard loud noise from {address}"); } }
templates/BreakInMurderTemplate.cs
public class BreakInMurderTemplate : IActionTemplate { public ActionResult Execute(ActionContext ctx) { var victim = ctx.GetTarget<Person>(); var home = ctx.State.GetAddress(victim.HomeAddressId); // Step 1: Get inside ForcedEntry.Execute(ctx); // Step 2: Find victim (they're in a sublocation) var victimRoom = FindPersonRoom(victim, home); Movement.MoveTo(ctx, victimRoom); // Step 3: Confrontation — does victim fight back? if (ctx.Random.Chance(0.6f)) { Violence.Fight(ctx, victim, victimRoom); // Chase? 40% chance victim flees if (victim.IsAlive && ctx.Random.Chance(0.4f)) { var fleeRoom = PickAdjacentRoom(home, victimRoom); Movement.ChaseFromTo(ctx, victim, victimRoom, fleeRoom); Violence.Fight(ctx, victim, fleeRoom); } } // Step 4: Kill Violence.Kill(ctx, victim, ctx.CurrentRoom); // Step 5: Leave (optional: clean up, search, take trophy) if (ctx.Random.Chance(0.2f)) Search.SearchFor(ctx, home, "valuables"); Movement.MoveTo(ctx, home.GetEntryPoint()); ForcedEntry.Exit(ctx); return new ActionResult { Duration = ctx.ElapsedTime }; } }

Pros

  • Full power of C# — any logic, any branching
  • Easy to debug, step through, unit test
  • Composable: Fight, MoveTo, ForcedEntry reused everywhere
  • IDE autocomplete, type safety, refactoring support

Cons

  • Adding a new template = writing C# code, recompiling
  • Non-programmers can't author templates
  • Harder to visualize the full "story" of a template at a glance

Approach B: Data-Driven — JSON/YAML Configs

Templates are config files. An interpreter engine runs them. Events are named operations.

File structure
src/simulation/actions/ engine/ ActionEngine.cs # Interprets template configs EventRegistry.cs # Maps event names to handlers TraceEmitter.cs handlers/ # C# handlers for each event type ForcedEntryHandler.cs MovementHandler.cs ViolenceHandler.cs data/templates/ # JSON config files break-in-murder.json burglary.json date-visit.json daily-routine.json
data/templates/break-in-murder.json
{ "name": "break_in_murder", "steps": [ { "event": "forced_entry", "target": "victim.home", "params": { "prefer": "window" } }, { "event": "move_to", "target": "victim.current_room" }, { "if": { "chance": 0.6 }, "then": [ { "event": "fight", "target": "victim", "location": "current_room" }, { "if": { "chance": 0.4, "condition": "victim.is_alive" }, "then": [ { "event": "chase", "from": "current_room", "to": "adjacent_room" }, { "event": "fight", "target": "victim", "location": "current_room" } ] } ] }, { "event": "kill", "target": "victim" }, { "if": { "chance": 0.2 }, "then": [ { "event": "search", "target": "victim.home", "for": "valuables" } ] }, { "event": "move_to", "target": "entry_point" }, { "event": "exit" } ] }

Pros

  • Templates are data — easy to read, diff, review
  • Could hot-reload without recompiling
  • Clear separation: engine logic vs. template authoring
  • Could build a visual editor someday

Cons

  • You're building a scripting language — conditionals, references, expressions
  • Debugging is painful (stack traces point to engine, not template)
  • Limited expressiveness — complex logic needs engine extensions
  • No type safety on template configs until runtime
  • Significant upfront engine investment

Approach C: Hybrid — C# Event Primitives + Fluent Builder API

Events are C# primitives. Templates compose them via a readable fluent API. Best of both?

File structure
src/simulation/actions/ events/ # Same reusable primitives as Approach A ForcedEntry.cs Movement.cs Violence.cs Social.cs Search.cs templates/ # Templates using fluent builder BreakInMurderTemplate.cs BurglaryTemplate.cs DateVisitTemplate.cs ActionSequence.cs # Fluent builder for composing events ActionContext.cs TraceEmitter.cs
ActionSequence.cs (the builder)
public class ActionSequence { public ActionSequence Do(Action<ActionContext> action) { ... } public ActionSequence MoveTo(string target) { ... } public ActionSequence Maybe(float chance, Action<ActionSequence> branch) { ... } public ActionSequence If(Func<ActionContext, bool> cond, Action<ActionSequence> then) { ... } public ActionSequence OneOf(params (float weight, Action<ActionSequence> branch)[] options) { ... } public ActionResult Execute(ActionContext ctx) { ... } }
templates/BreakInMurderTemplate.cs
public class BreakInMurderTemplate : IActionTemplate { public ActionResult Execute(ActionContext ctx) { var victim = ctx.GetTarget<Person>(); return new ActionSequence() .Do(ForcedEntry.Enter) .MoveTo("victim.current_room") .Maybe(0.6f, s => s .Do(c => Violence.Fight(c, victim)) .If(c => victim.IsAlive, s2 => s2 .Maybe(0.4f, s3 => s3 .Do(c => Movement.ChaseTo(c, victim, "adjacent_room")) .Do(c => Violence.Fight(c, victim)) ) ) ) .Do(c => Violence.Kill(c, victim)) .Maybe(0.2f, s => s .Do(c => Search.SearchFor(c, "valuables")) ) .MoveTo("entry_point") .Do(ForcedEntry.Exit) .Execute(ctx); } }

Pros

  • Reads like a script — the "story" is visible at a glance
  • Full C# power when you need it (lambdas, conditions)
  • Same reusable event primitives as Approach A
  • Type-safe, debuggable, IDE-friendly
  • Builder API is thin — not building a scripting language
  • Easy to add new control flow (OneOf for weighted random, Repeat, etc.)

Cons

  • Still requires recompilation for new templates
  • Fluent API has a learning curve (lambdas within lambdas)
  • Builder needs to handle time accumulation, trace collection, context threading
