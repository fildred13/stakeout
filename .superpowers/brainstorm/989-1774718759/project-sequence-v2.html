Simulation Overhaul — Revised Project Sequence

Sequential, no parallelization. Unified NPC brain + action templates. Player UI included.

Key Revision: Intents = Action Templates

There is no separate "intent" layer and "action template" layer. Everything an NPC does is an action template being executed. The NPC brain selects which action to run. Actions can push sub-actions (that's the stack). The "routine" is the brain's default action selection logic — not a separate data structure.

NPC Brain (selector) "What should I do right now?" → evaluates: obligations, drives, defaults, interrupts → picks an action template │ Action Template (executor) "How do I do it?" → runs composable steps, emits traces → can push sub-actions onto the stack → completes → brain selects next action │ Action Library (catalog) Simple: IdleAtHome, Sleep, GoForARun, EatAtDiner Moderate: BuyDrugs, GoOnDate, Commute, WorkShift Complex: BreakInMurder, Heist, Burglary All use the same machinery. "Go for a run" and "do a heist" differ in complexity, not in kind.

Revised Project Sequence

Each project gets: design spec → implementation plan → implementation. All sequential.

Project 1

Location Hierarchy & Address Templates

Replace the sublocation graph with Address → Location → Sub-Location. Composable address type templates. Doors/access points as location properties. Migrate all generators.

Delivers
  • New Address, Location, SubLocation entities with tag system
  • Door/AccessPoint as mutable objects on locations
  • Address type template system (composable location types)
  • Migrated generators: SuburbanHome, Apartment, Diner, Office, Park, DiveBar
  • Remove: SublocationGraph, SublocationConnection, BFS pathfinding, TraversalContext
Why first

Everything references locations. This is the data model everything else builds on.

↓
Project 2

Fixtures & Trace System

Fixtures (stable objects from templates, can contain traces) and traces (dynamic evidence from events). Trace decay. TraceEmitter API. Investigation query merging both systems.

Delivers
  • Fixture entity with container semantics
  • Fixture definitions in address type templates
  • Redesigned Trace entity (marks, items, sightings, records, fingerprints)
  • Trace decay over time
  • TraceEmitter API (what action templates will call)
  • Investigation query interface (merged fixture + trace discovery)
  • Remove: FingerprintService
Why second

The trace system is what actions produce. Need the output system before building the action engine.

↓
Project 3

NPC Brain & Action Engine

The unified NPC brain: action selection (obligations + drives + defaults) and action execution (fluent builder, composable event primitives, sub-action stacking). The action library of reusable templates. Event log for history. This is the big rewrite.

Delivers
  • Action stack (push/pop sub-actions, priority-based interruption)
  • Brain: obligation/drive/default evaluation → action selection
  • ActionSequence fluent builder (Do, MoveTo, Maybe, If, OneOf)
  • ActionContext (state bag threaded through events)
  • Event primitives: ForcedEntry, Movement, Violence, Social, Search
  • Initial action library: Sleep, IdleAtHome, Commute, WorkShift, EatOut, GoForARun
  • NPC travel between addresses (triggered by action, not pre-scheduled)
  • Fast-forward (routine NPCs as time→location, active NPCs step-through)
  • Event log for history/investigation
  • Trait → drive mapping (initial set: runner, addict, devout, etc.)
  • Remove: ScheduleBuilder, ObjectiveResolver, DailySchedule, PersonBehavior, TaskResolver, all decomposition strategies
Why third

The brain is the core engine. Needs locations (P1) and traces (P2) as foundations. Everything after this builds on the brain.

↓
Project 4

Business Entities & Staffing

Businesses as first-class entities. Operating hours, shift definitions, day-of-week variation. NPC generation fills shifts. Job obligations feed into the brain's action selection.

Delivers
  • Business entity (operating hours, shift slots, staffing rules)
  • Business type templates (diner, bar, office, bank, etc.)
  • Shift ↔ NPC assignment during generation
  • Business ↔ Address linkage
  • Day-of-week and shift variation
  • Job obligation integration with NPC brain (shifts → work action selection)
Why fourth

Businesses provide the data that job obligations consume. Brain (P3) must exist first so we can wire shifts into action selection.

↓
Project 5

Groups & Coordinated Actions

Group entity with shared action. Members join/leave. Split/Regroup for parallel sub-actions. Same system for a date and a heist.

Delivers
  • Group entity (members, shared state, shared vehicle/loot)
  • Group action (shared action pushed onto all members' stacks)
  • Join/Leave/Disband lifecycle
  • Split/Regroup for parallel sub-actions
  • Member interruption handling
  • Group action templates: GoOnDate, CriminalMeeting
Why fifth

Groups are a coordination layer on top of the brain + action engine.

↓
Project 6

History Backfill & NPC Spawning

New NPCs spawn with plausible history. Routine-based trace backfill for general population. Event-level backfill for crime NPCs.

Delivers
  • Routine-based backfill (derive traces from routine over N days)
  • Event-level backfill for crime NPCs (planted history)
  • Late-spawning NPC integration
Why sixth

Backfill uses routines (P3), traces (P2), and optionally groups (P5).

↓
Project 7

Crime Templates & Scenarios

Rebuild the crime system on the new engine. Crime scenarios use action templates, groups, and trace generation. Start with murder, expand from there.

Delivers
  • Crime scenario system (replaces CrimeGenerator + ICrimeTemplate)
  • Murder scenario with variable methods and rich trace generation
  • Crime NPC casting from population
  • Drive injection (crime gives NPCs new drives)
  • Group integration for multi-person crimes
Why seventh

Crimes exercise the full stack: locations, traces, brain, actions, groups, backfill.

↓
Project 8

Player Interaction UI

The player-facing screens that bring it all together. Address navigation (enter, investigate, break in). Investigation mechanic (timed evidence discovery). Stakeout event stream with trigger conditions. Fixture interaction (hack computer, search filing cabinet). Evidence board integration.

Delivers
  • Address screen (exterior view, action options based on address type/state)
  • Interior navigation (location list, sublocation list for applicable locations)
  • Door interaction (knock, try keys, pick lock, break down, dust for prints)
  • Investigation mechanic (timed discovery of fixtures + traces, difficulty-weighted)
  • Fixture interaction screens (computer hacking, searching containers, etc.)
  • Stakeout event stream with trigger conditions and auto-realtime
  • Evidence board integration (right-click to add, red/white text coloring)
Why last

The UI consumes everything: locations, fixtures, traces, NPC presence, event streams. It's the capstone that makes it all playable.
