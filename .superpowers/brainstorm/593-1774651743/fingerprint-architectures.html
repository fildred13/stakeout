Fingerprints are Trace entities, but surfaces (doors/items) maintain a list of trace IDs for fast access

CONNECTIONS:
Person → creates → Trace
Trace → references → Surface (via Data dict)
Surface → indexes → Trace IDs ← bidirectional
EvidenceBoard → queries → Trace
ActionExecutor → creates → Trace + updates → Surface

Trade-off Comparison

Dimension A: Trace-Only B: Hybrid
Smudging speed Scan all traces filtered by surface Direct list on surface — O(n) where n = fingerprints on that surface
Complexity Single source of truth, no sync Must keep surface index in sync with Trace lifecycle
Existing pattern fit Matches current Trace usage (crime system) Matches composable property pattern (LockableProperty, BreakableProperty)
Extensibility New surface types = query changes only New surface types = add FingerprintSurface property, automatic indexing
Investigation query Must filter all traces to find what's on a door "Show fingerprints on this door" = read surface directly
Scale concern Could get slow with thousands of traces Index overhead is trivial (list of ints)

Recommendation: B (Hybrid)

The hybrid approach matches the codebase's existing composable-property pattern (like LockableProperty) and makes the smudging mechanic natural — you just iterate the fingerprints on that specific surface. The sync cost is minimal since fingerprint creation and removal happen in one place (ActionExecutor).
