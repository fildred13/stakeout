# Location Hierarchy

## Purpose
Models the physical world as a strict hierarchy: World → City → Address → Location → SubLocation. Locations represent distinct areas within an address (floors, yards, rooms), while SubLocations are optional subdivisions (bedroom, kitchen within an apartment unit). Address templates generate interiors compositionally using reusable LocationBuilders.

## Key Files
| File | Role |
|------|------|
| `src/simulation/entities/Address.cs` | Address entity — holds CityId, AddressType, list of LocationIds |
| `src/simulation/entities/Location.cs` | Location entity — Id, AddressId, Name, Tags, Floor, UnitLabel, SubLocationIds, AccessPoints |
| `src/simulation/entities/Sublocation.cs` | SubLocation entity — Id, LocationId, Name, Tags, AccessPoints (file named for Windows case-insensitivity) |
| `src/simulation/entities/AccessPoint.cs` | AccessPoint class + AccessPointType/LockMechanism enums — doors, windows, gates with lock state |
| `src/simulation/entities/City.cs` | City entity — AddressIds list, AirportAddressId |
| `src/simulation/addresses/IAddressTemplate.cs` | Interface: Generate(address, state, random) — creates Locations/SubLocations for an address |
| `src/simulation/addresses/AddressTemplateRegistry.cs` | Static registry mapping AddressType → IAddressTemplate |
| `src/simulation/addresses/LocationBuilders.cs` | Reusable static helpers: CreateLocation, CreateSubLocation, ExteriorParkingLot, ApartmentUnit, etc. |
| `src/simulation/addresses/*Template.cs` | Seven templates: SuburbanHome, Diner, DiveBar, ApartmentBuilding, Office, Park, Airport |

## How It Works
Each City contains Addresses placed on a CityGrid. When an address is generated, `AddressTemplateRegistry` looks up the template for its AddressType and calls `Generate()`. Templates use `LocationBuilders` helpers to create Locations and SubLocations, adding them to `SimulationState.Locations` and `SimulationState.SubLocations` dictionaries.

AccessPoints (doors, windows, gates) are properties of the Location or SubLocation they guard — not graph edges. A locked front door is an AccessPoint on the "Interior" Location with `IsLocked = true` and a `KeyItemId`. There is no pathfinding graph; movement between locations will be handled by future scheduling/behavior systems.

Multi-city is supported: Boston and NYC each get their own CityGrid, addresses, and an Airport address for inter-city travel. `SimulationState` provides query helpers: `GetLocationsForAddress`, `GetSubLocationsForLocation`, `FindLocationByTag`, `GetAddressesForCity`, `GetCityForAddress`.

## Key Decisions
- **Strict 2-level hierarchy (Location → SubLocation):** No recursive nesting. Apartment units are Locations; rooms within are SubLocations. Simple and sufficient.
- **AccessPoints on what they guard:** Doors belong to rooms, not to the space between rooms. Eliminates the graph edge model and artificial "at a door" states.
- **Composable templates + LocationBuilders:** Templates call shared helpers (ExteriorParkingLot, ApartmentUnit, Restroom) to avoid duplication across address types.
- **Tags for behavioral queries:** String arrays on Locations/SubLocations (e.g., "exterior", "residential", "entrance") replace hardcoded type checks.

## Connection Points
- **Generated by:** LocationGenerator calls templates during world setup; CityGenerator creates Cities and places Addresses on grids
- **Referenced by:** Person.CurrentLocationId/CurrentSubLocationId, Player.CurrentLocationId track where entities are
- **Display:** CityView reads CityGrids; BlueprintView and GraphView read Location/SubLocation structure (currently stubbed with TODOs)
- **Scheduling (TODO):** Future scheduling system will use tags and AccessPoints to place people in rooms and handle movement
