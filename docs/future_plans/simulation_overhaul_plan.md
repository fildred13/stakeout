# Simulation Overhaul — Master Plan

This document captures the design decisions and project sequence for a major overhaul of the NPC simulation engine. It serves as the "constitution" for all sub-projects — each sub-project gets its own design spec, implementation plan, and implementation cycle, but must align with the decisions here.

This plan originated from `docs/future_plans/sublocations_rethink.md` and was developed through collaborative brainstorming.

## Design Decisions

### Location Hierarchy

**World → City → Address → Location → (optional) Sub-Location.**

Strict 2-level nesting below Address, not recursive. Addresses are composed of Locations (exterior parking lot, security room, Unit 1A, etc.). Some Locations have Sub-Locations (an `apartment_unit` Location has bedroom, living room, kitchen, balcony Sub-Locations; an `office_floor` has cubicles, manager's office, break room).

Tags (`exterior`, `publicly_accessible`, etc.) handle behavioral variations. Composable location types (ExteriorParkingLot, ApartmentUnit, etc.) bring their own sub-locations, tags, and rules — reusable across address types.

**Replaces:** The current sublocation graph with BFS pathfinding, SublocationConnection edges, and TraversalContext. The player never navigates room-to-room — they interact through action menus.

### Doors & Access Points

Doors are real objects with mutable state (locked/unlocked, broken/repaired), but they're **properties of the Location or Sub-Location they guard** — not edges in a graph. Address type templates define what doors, windows, etc. exist as part of their location composition. No more BFS pathfinding or sublocation graph traversal.

### Fixtures vs Traces

**Separate systems, merged at query time for investigation.**

- **Fixtures** are stable elements defined by address type templates: computers, mailboxes, safes, trash cans, filing cabinets. They define what you *can* interact with at a location. Fixtures are not serialized in saves — they're regenerated from templates.
- **Traces** are dynamic evidence generated by NPC actions and events: fingerprints, blood, receipts, witness reports. They define what you *find* when you interact. Traces are serialized in saves and can decay over time.
- **Fixtures can contain traces.** The computer is always there (fixture), but the emails on it are traces generated by the NPC's life. Same for mailboxes (contain letters), trash cans (contain receipts), answering machines (contain messages).

The investigation mechanic queries both systems and merges them into a single discovery stream for the player, weighted by difficulty and relevance to the player's evidence board.

### NPC Brain — Intent Stack + Action Engine (Unified)

**There is no separate "intent" layer and "action template" layer.** Everything an NPC does is an action template being executed. The NPC brain selects which action to run. Actions can push sub-actions (that's the stack). The "routine" is the brain's default action selection logic — not a separate data structure.

**Replaces:** The pre-computed 1440-slot daily schedule. The old system plans the entire day ahead of time, then struggles when the future doesn't cooperate (interrupts, arrests, player intervention). The intent stack is reactive by design — push to interrupt, pop to resume.

**Brain — Action Selection:**

The brain evaluates three layers to decide "what should I do right now?":

1. **Obligations** — things the NPC must do: job shift schedule (from their employer), sleep (biological, timing adapts to work schedule).
2. **Drives** — things the NPC wants to do, based on traits and current state: "drug addict" + no drugs → seek dealer; "avid runner" → run at noon; dating someone → occasional date. Drives are emergent from traits, inventory, relationships, and time/day.
3. **Defaults** — what happens when nothing else applies: be at home.

Routines are **not hardcoded schedules** — they're emergent behavior from these layers. A bartender who works nights and has the "runner" trait will naturally sleep late, run at noon, and work evenings. Their schedule varies by day of week (days off), inventory state (needs drugs?), relationships (date night?), etc.

**Action Engine — Action Execution:**

Actions are executed via a **fluent builder API** (ActionSequence) with composable event primitives:

- Event primitives: ForcedEntry, Movement, Violence, Social, Search, etc. — reusable building blocks.
- ActionSequence builder: `.Do()`, `.MoveTo()`, `.Maybe()`, `.If()`, `.OneOf()` — compose events into readable "stories."
- Actions produce traces via TraceEmitter as they execute.
- Actions can push sub-actions onto the stack — that's how complex templates decompose.

**"Go for a run" and "do a heist" use the same machinery.** They differ in complexity, not in kind. The action library is a catalog of templates ranging from simple (IdleAtHome, Sleep, GoForARun) to complex (BreakInMurder, Heist).

**Fast-forward model:** Routine NPCs (hundreds/thousands) are essentially free — the routine is a function `(time, day) → location`, no per-tick simulation. Only NPCs with active non-routine actions (a handful of criminals) need step-through resolution.

### Groups & Coordinated Actions

**Explicit Group entity with shared action.** Members join/leave. The Group owns shared state (vehicle, loot). Group actions manage shared timelines with Split/Regroup for parallel sub-actions.

A Group of 2 people on a date and a Group of 4 people on a heist use the same system. Individual actions suspend while grouped and resume on disband.

**Example — Heist flow:**
1. Each member gets a "go to meetup point" action (individual).
2. On arrival, they join the Group. Individual actions suspend.
3. Group action takes over: drive together, wait, split for roles (driver/gunman/safecracker), regroup, drive to lot.
4. Group disbands. Individual actions resume.
5. Post-disband solo actions (driver stashes money). Delayed follow-up (bagman picks up hours later).

If a member is interrupted (arrested, killed), the group detects it and can branch to contingency — no schedule invalidation needed.

### Trace Generation Model

**Declarative, not physical.** Traces come from two sources:

1. **Facts** — residency generates bulk fingerprints and personal items. "This person lives here" → their fingerprints are throughout the home, their mail is in the mailbox, their prescription bottles are in the bathroom. No per-door traversal simulation needed.
2. **Actions** — action templates emit traces as part of execution. A date visit emits visitor fingerprints in scattered rooms, an extra wine glass, maybe a forgotten item. A break-in murder emits broken window, blood trail, killer's fingerprints, struggle marks. The action template defines what traces to generate, with variability (randomized, conditional).

Traces can **decay over time** — a visitor's fingerprints fade after a few days, but a resident's are constantly refreshed.

### History Backfill

When new NPCs enter the simulation, they get a plausible past:

- **Routine-based backfill** for general population: derive traces from the routine pattern over N days. Cheap — no simulation, just "this person has been living at this address and working at this job, generate the expected traces."
- **Event-level backfill** for crime-related NPCs: planted past events (meetings with the crime boss, purchases, suspicious visits) that generate specific traces at specific locations.

### Businesses

First-class entities with operating hours, shift definitions, and staffing requirements. NPC generation fills shift slots. Once spawned, NPCs show up for work via their job obligation in the brain's action selection. The business is the source of truth for "what shifts exist" — it doesn't actively manage NPCs at runtime.

Day-of-week awareness: a bar might be open every night but closed Mondays. An office runs 9-5 weekdays.

### Performance Model

- Hundreds to low thousands of NPCs active at any time, but only a handful of criminals doing "interesting" things.
- Routine NPCs: free to fast-forward (time → location function).
- Active NPCs: step-through resolution during fast-forward (small number).
- Stakeout: event stream of arrivals/departures at an address. Player can set trigger conditions ("go to realtime if I see any of these people, or people with these characteristics").
- Clock is continuous — no "jump to time" mechanic; time accelerates smoothly.

## Project Sequence

All sequential. Each project gets its own design spec → implementation plan → implementation cycle.

### Project 1: Location Hierarchy & Address Templates

Replace the sublocation graph with Address → Location → Sub-Location. Composable address type templates. Doors/access points as location properties. Migrate all generators.

**Delivers:**
- New Address, Location, SubLocation entities with tag system
- Door/AccessPoint as mutable objects on locations
- Address type template system (composable location types)
- Migrated generators: SuburbanHome, Apartment, Diner, Office, Park, DiveBar
- Remove: SublocationGraph, SublocationConnection, BFS pathfinding, TraversalContext

**Why first:** Everything references locations. This is the data model everything else builds on.

### Project 2: Fixtures & Trace System

Fixtures (stable objects from templates, can contain traces) and traces (dynamic evidence from events). Trace decay. TraceEmitter API. Investigation query merging both systems.

**Delivers:**
- Fixture entity with container semantics
- Fixture definitions in address type templates
- Redesigned Trace entity (marks, items, sightings, records, fingerprints)
- Trace decay over time
- TraceEmitter API (what action templates will call)
- Investigation query interface (merged fixture + trace discovery)
- Remove: FingerprintService

**Why second:** The trace system is what actions produce. Need the output system before building the action engine.

### Project 3: NPC Brain & Action Engine

The unified NPC brain: objective-driven daily planning and action execution. All NPC behavior flows through one system — objectives produce actions, the brain schedules them into a daily plan, and the action runner executes them.

**Design decisions made during P3 brainstorming:**
- **Unified objective model.** The original three-layer brain (obligations/drives/defaults) was simplified. Everything is an objective with a priority. The daily planner sorts objectives by priority and schedules greedily. A runner's hobby and a killer's murder use the same algorithm — they differ in priority, not in kind.
- **Objective composition.** Objectives can have child objectives. A parent delegates a phase to a child (e.g., HeistObjective → RecruitCrew → FindCandidate). Multiple children can be active in parallel. This enables reusable building blocks for crime templates in P7.
- **Daily planning, not per-tick evaluation.** The brain plans the full day on wake-up (and re-plans after interrupts, when added). Between planning moments, the NPC executes the plan sequentially.
- **Travel is the engine's job.** Actions define a destination + activity. The engine handles inter-address travel universally. Actions never contain "go to X" boilerplate — each action is self-contained as "what to do once there."
- **No fast-forward optimization yet.** All NPCs use the same action stack path. The time→location optimization for routine NPCs is deferred until there's a real performance need and the action library is mature enough to define "routine."
- **Job-related features deferred to P4.** WorkShift, Commute, job assignment in PersonGenerator, and DoorLockingService are commented out with TODO markers. P3 builds the clean base; P4 restores job functionality via Business entities.
- **Only build primitives P3's actions use.** WaitAction and MoveToAction (intra-address). ForcedEntry, Violence, Social, Search are deferred to P5+/P7.

**Delivers:**
- Objective base class (priority, state, phases, child objectives, GetActionsForToday)
- NpcBrain daily planner (sort objectives by priority, schedule greedily, fill gaps)
- DayPlan (ordered action list with time slots for today)
- ActionRunner (per-frame: handle travel, tick actions, advance plan)
- IAction interface + ActionSequence fluent builder (Do, MoveTo, Wait, Maybe, If)
- ActionContext (state bag: Person, SimulationState, TraceEmitter, EventJournal, Random)
- Event primitives: WaitAction, MoveToAction (intra-address)
- Initial objective library: SleepObjective, GoForARunObjective, EatOutObjective, IdleAtHome (gap-filler)
- Trait system: traits on Person, TraitDefinitions registry mapping traits → objectives (runner, foodie)
- NPC travel between addresses (engine-level, reusing existing MapConfig/TravelInfo)
- Event log extensions (ActivityStarted, ActivityCompleted event types)
- Debug inspector updates (day plan view, objective list, current activity display)
- Remove: ScheduleBuilder, DailySchedule, TaskResolver, ObjectiveResolver, all decomposition strategies
- Gut with TODO: PersonBehavior (→ ActionRunner), job assignment in PersonGenerator (→ P4), DoorLockingService (→ P4)

**Why third:** The brain is the core engine. Needs locations (P1) and traces (P2) as foundations. Everything after this builds on the brain.

**Full design spec:** `docs/superpowers/specs/2026-03-29-npc-brain-action-engine-design.md`

### Project 4: Business Entities & Staffing

Businesses as first-class entities. Operating hours, shift definitions, day-of-week variation. NPC generation fills shifts. Job obligations feed into the brain's action selection.

**Delivers:**
- Business entity (operating hours, shift slots, staffing rules)
- Business type templates (diner, bar, office, bank, etc.)
- Shift ↔ NPC assignment during generation
- Business ↔ Address linkage
- Day-of-week and shift variation
- Job obligation integration with NPC brain (WorkShiftObjective, CommuteAction)
- Restore job assignment in PersonGenerator (commented out in P3 with TODO: Project 4)
- DoorLockingService implementation (commented out in P3 with TODO: Project 4)

**Why fourth:** Businesses provide the data that job obligations consume. Brain (P3) must exist first so we can wire shifts into action selection.

### Project 5: Groups & Coordinated Actions

Group entity with shared action. Members join/leave. Split/Regroup for parallel sub-actions. Same system for a date and a heist.

**Delivers:**
- Group entity (members, shared state, shared vehicle/loot)
- Group action (shared action pushed onto all members' stacks)
- Join/Leave/Disband lifecycle
- Split/Regroup for parallel sub-actions
- Member interruption handling
- Group action templates: GoOnDate, CriminalMeeting

**Why fifth:** Groups are a coordination layer on top of the brain + action engine.

### Project 6: History Backfill & NPC Spawning

New NPCs spawn with plausible history. Routine-based trace backfill for general population. Event-level backfill for crime NPCs.

**Delivers:**
- Routine-based backfill (derive traces from routine over N days)
- Event-level backfill for crime NPCs (planted history)
- Late-spawning NPC integration

**Why sixth:** Backfill uses routines (P3), traces (P2), and optionally groups (P5).

### Project 7: Crime Templates & Scenarios

Rebuild the crime system on the new engine. Crime scenarios use action templates, groups, and trace generation. Start with murder, expand from there.

**Delivers:**
- Crime scenario system (replaces CrimeGenerator + ICrimeTemplate)
- Murder scenario with variable methods and rich trace generation
- Crime NPC casting from population
- Drive injection (crime gives NPCs new drives)
- Group integration for multi-person crimes

**Why seventh:** Crimes exercise the full stack: locations, traces, brain, actions, groups, backfill.

### Project 8: Player Interaction UI

The player-facing screens that bring it all together. Address navigation, investigation, stakeout, fixture interaction, evidence board integration.

**Delivers:**
- Address screen (exterior view, action options based on address type/state)
- Interior navigation (location list, sublocation list for applicable locations)
- Door interaction (knock, try keys, pick lock, break down, dust for prints)
- Investigation mechanic (timed discovery of fixtures + traces, difficulty-weighted)
- Fixture interaction screens (computer hacking, searching containers, etc.)
- Stakeout event stream with trigger conditions and auto-realtime
- Evidence board integration (right-click to add, red/white text coloring)

**Why last:** The UI consumes everything: locations, fixtures, traces, NPC presence, event streams. It's the capstone that makes it all playable.
